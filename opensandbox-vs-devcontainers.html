OPENSANDBOX ✕ DEV CONTAINERS

Cùng dùng container.
Giải hai bài toán khác nhau.

Cả hai đều đặt code vào trong container để cô lập — nên dễ tưởng là thay thế được cho nhau. Nhưng chúng đứng ở hai tầng khác nhau và trả lời hai câu hỏi khác nhau. Trang này phân định rạch ròi, nhìn qua lăng kính bạn quan tâm: chạy AI agent.

Câu trả lời ngắn gọn

Khác bài toán, trùng cơ chế. Cùng là container, nhưng một cái là môi trường để bạn làm việc, cái kia là nền tảng để ứng dụng vận hành workload.

Dev Container

Lo bài toán môi trường phát triển nhất quán cho một người, một repo. Diễn ra ở dev-time, do IDE điều khiển. Cô lập chỉ là lợi ích phụ.

OpenSandbox

Lo bài toán chạy workload cô lập theo chương trình, ở quy mô lớn cho một ứng dụng/agent. Diễn ra ở runtime, do API điều khiển. Cô lập an ninh là mục tiêu chính.

01 — Mỗi cái là gì

Định nghĩa lại cho rõ

Dev Container

Một căn phòng làm việc dựng sẵn

"Làm sao để tôi (và máy đồng đội) có cùng một bộ công cụ nhất quán?"

Sinh ra từ VS Code Remote Containers. Một file devcontainer.json mô tả base image, extensions, lệnh post-create, mount… để biến "chạy được trên máy tôi" thành "chạy được trong một container đã định nghĩa".

  • Một container lâu dài, gắn với workspace/IDE của bạn
  • Repo của bạn được mount thẳng vào — container có toàn quyền với code
  • Thường forward SSH agent, git credential, đôi khi cả Docker socket
  • Mục tiêu: tái lập & tiện lợi, không phải chống code độc hại
OpenSandbox

Một nhà máy dựng phòng theo yêu cầu

"Làm sao ứng dụng của tôi tạo & chạy hàng loạt môi trường cô lập, từ code, ở quy mô?"

Một control plane (server FastAPI) + data plane (container của bạn có daemon execd tiêm vào). Bạn gọi API để create → run lệnh/code/file → snapshot/pause → kill, với TTL tự hết hạn.

  • N môi trường ephemeral, tạo/huỷ theo chương trình
  • Điều khiển qua REST API, SDK 5 ngôn ngữ, CLI osb, MCP
  • Egress policy, secure runtime (gVisor/Kata/Firecracker), Credential Vault
  • Chạy trên Docker (local) hoặc Kubernetes (BatchSandbox, Pool) ở quy mô

04 — Lăng kính AI agent

Cùng chữ "agent", hai ngữ cảnh

Đây là chỗ dễ nhầm nhất. Cùng nói "chạy AI agent trong container", nhưng vai trò của agent và của bạn hoàn toàn khác.

Devcontainer cho agent

Agent là đồng đội trên repo của bạn

Bạn — một lập trình viên — muốn Claude Code / Cursor làm việc trong một căn phòng nhất quán & an toàn ngay trên code của bạn. Một cái, lâu dài, lấy con người làm trung tâm. Agent sửa đúng codebase đó.

"Cho trợ lý code của tôi một chỗ an toàn, đồng nhất để sửa repo này — đừng đụng vào host."
OpenSandbox cho agent

Agent là workload sản phẩm của bạn chạy

Sản phẩm của bạn chạy code/agent mà bạn không kiểm soát hoặc không tin tưởng, có thể hàng nghìn lần, cô lập lẫn nhau và khỏi hạ tầng. Ephemeral, multi-tenant, ưu tiên an ninh.

"SaaS của tôi cho phép người dùng chạy code AI sinh ra" · "Tôi vận hành một fleet RL training" · "Nền tảng agent của tôi spawn sub-agent."
05 — Chọn cái nào

Dùng cái nào, khi nào

Chọn Dev Container khi…

  • Bạn/team phát triển trên một (vài) repo cụ thể
  • Muốn toolchain tái lập, onboard nhanh "clone là code được"
  • Muốn trợ lý AI có chỗ làm việc đồng nhất, an toàn trên code của bạn
  • Không cần tạo môi trường bằng code, không cần scale

Chọn OpenSandbox khi…

  • Bạn xây sản phẩm/nền tảng chạy code hoặc agent bằng chương trình
  • Cần multi-tenant và scale tới nhiều môi trường cô lập đồng thời
  • Cần kiểm soát egress mạng, cô lập mạnh cho code không tin cậy
  • Cần snapshot / pause-resume / pool / điều phối batch & RL
06 — Không loại trừ nhau

Và chúng kết hợp được

Hai cái không đối nghịch. Bạn chạy được một coding agent bên trong một OpenSandbox — trong repo còn có sẵn ví dụ examples/claude-code. Hiểu theo cách này: OpenSandbox chính là phiên bản công nghiệp, điều khiển bằng API, multi-tenant của thứ mà devcontainer làm cho một người. Devcontainer là bản tĩnh, khai báo, một-instance; OpenSandbox là bản động, lập trình được, nhiều-instance.

Dev Container

Như một xưởng cá nhân: đồ nghề của bạn, bàn làm việc của bạn, set up một lần, bạn làm việc ở đó.

OpenSandbox

Như một nhà máy dựng & tháo xưởng theo yêu cầu qua API — có bảo vệ, kiểm soát đường cấp (egress), chạy hàng trăm xưởng cùng lúc.

Một cách nhớ: devcontainer : OpenSandbox ≈ docker run trên laptop của tôi : một nền tảng compute cô lập cho ứng dụng. Devcontainer trả lời câu hỏi dev-time (tôi phát triển trong môi trường nào?); OpenSandbox trả lời câu hỏi runtime (ứng dụng của tôi thực thi workload động/không tin cậy ở đâu?).

07 — Đã chạy thật

Kiểm chứng trên máy của bạn

Không phải lý thuyết — đây là output thật khi dựng OpenSandbox trên OrbStack (Docker 29.4.0) ngay trong phiên này.

orbstack · opensandbox · localhost:8080
$ uvx opensandbox-server            # control plane lên ở 127.0.0.1:8080, Swagger ở /docs

$ osb sandbox create --image python:3.12 --timeout 30m
  → id: c8ffefe3-0ea0-4111-963c-b2369a100965   status: running

$ docker ps
  sandbox-c8ffefe3...   python:3.12   Up   0.0.0.0:47713->8080, ...->44772  # container thật + execd

$ osb command run <id> -- python -c "import sys,platform; print(sys.version, platform.machine())"
  Python 3.12.13   aarch64

$ osb command run <id> -- python -c "import math; print(math.factorial(10))"
  3628800

$ osb command run <id> -- ls /Users          # thử nhìn ra host macOS
  ls: cannot access '/Users': No such file or directory   → cô lập khỏi host ✓

$ osb sandbox kill <id>
  ○ terminated   # ephemeral: tạo & huỷ theo chương trình

Lưu ý nhỏ ghi nhận được: image opensandbox/execd có trên Docker Hub nhưng đôi khi trả 401 tạm thời khi pull đồng thời — pull trước một lần là tạo sandbox chạy ngay. CLI dùng lệnh osb sandbox kill (không phải delete), và field id trong JSON là id.
