OpenSandbox · Chốt hạ & Hướng dẫn Docker

Bạn có thật sự cần OpenSandbox không?

OpenSandbox là hạ tầng để phần mềm của bạn tạo & chạy môi trường cô lập theo chương trình, ở quy mô. Sự thật thẳng thắn: phần lớn mọi người sẽ không cần nó. Trang này giúp bạn tự loại mình ra trong 30 giây — và nếu bạn thật sự cần, có sẵn hướng dẫn chạy bằng Docker đã kiểm chứng.

✓ Đã dựng & chạy thật trên macOS + OrbStack (Docker 29.4.0) trong phiên này
01 — Bài toán nó giải

Nó tồn tại để làm gì

Tinh thần cốt lõi

Khi phần mềm của bạn — chứ không phải bạn ở IDE — cần dựng nhiều môi trường thực thi cô lập, theo yêu cầu, để chạy code/agent không kiểm soát hoặc không tin tưởng, rồi huỷ đi.

  • Giao diện theo chương trình: REST API · SDK (Python/JS/Java/C#/Go) · CLI osb · MCP — tạo → chạy lệnh/code/file → snapshot/pause → kill, có TTL tự hết hạn.
  • Cô lập an ninh là mục tiêu chính: egress network policy, secure runtime (gVisor/Kata/Firecracker), Credential Vault, drop capabilities.
  • Quy mô: chạy trên Docker (local/single-host) hoặc Kubernetes (BatchSandbox, Pool pre-warm) cho hàng loạt sandbox đồng thời.
02 — Trước khi đọc tiếp

Bạn có nằm trong số cần nó không?

Đọc một cột bên dưới và dừng lại đúng lúc. Nếu bạn rơi vào cột trái, đừng tốn thời gian với phần còn lại của trang — có công cụ phù hợp hơn cho bạn.

?

Câu hỏi gốc để phân loại: ai là người tạo ra môi trường — bạn (ngồi ở IDE) hay code của bạn (gọi qua API)?

Dừng lại

Bạn KHÔNG cần OpenSandbox

  • Bạn muốn cô lập môi trường dev cho từng project (Java version, Maven config riêng) để khỏi sửa PATH/ENV liên tục
  • Bạn muốn Claude Code / Codex làm việc an toàn, đồng nhất trên repo của bạn
  • Bạn chỉ chạy code của chính mình, trên một máy, một project một lúc
  • Bạn không tạo môi trường "bằng code" và không cần scale hay multi-tenant

→ Dùng Dev Container (hoặc venv/SDKMAN/docker run). Đây là lựa chọn đúng cho nhu cầu của bạn. Đóng tab này được rồi.

Đọc tiếp

Bạn CÓ THỂ cần OpenSandbox

  • Bạn xây sản phẩm chạy code do người dùng hoặc model sinh ra (online judge, tính năng "Run", AI code-execution)
  • Bạn chạy/eval agent trên nhiều repo tự động, song song (kiểu SWE-bench) — mỗi cái một sandbox sạch rồi tear down
  • Bạn cần multi-tenant: cô lập code của nhiều người khỏi nhau và khỏi hạ tầng
  • Bạn cần egress control / pause-resume / snapshot / pool / scale trên Kubernetes

→ Tiếp tục xuống phần Yêu cầu & Hướng dẫn Docker bên dưới.

Quy tắc một dòng

Môi trường do BẠN tạo qua IDE → devcontainer. Do CODE của bạn tạo qua API, hàng loạt, ephemeral → OpenSandbox.

03 — Yêu cầu môi trường

Cần gì trên máy

Tối thiểu để chạy bản local với runtime Docker. Đây là cấu hình đã kiểm chứng trong phiên này.

✓ Docker Engine 20.10+

Bắt buộc cho chạy local. OrbStack hoạt động tốt — đã test trên Server 29.4.0. Docker Desktop cũng được.

✓ Python 3.10+

Nếu máy thấp hơn (vd macOS mặc định 3.9.6) thì không cần lo — uv sẽ tự tải Python 3.10 khi chạy.

✓ uv khuyến nghị

Quản package & Python tự động. Cài: brew install uv. Đã dùng uv 0.11.25. Cho phép chạy uvx không cần cài sẵn.

✓ OS & dung lượng

macOS / Linux / Windows + WSL2. Cần vài trăm MB cho image python:3.12 và opensandbox/execd.

04 — Hướng dẫn Docker

Chạy thử bằng Docker, từng bước

Đúng trình tự đã chạy thành công trên máy macOS + OrbStack. Copy chạy lần lượt là lên.

1

Cài uv & bật Docker

uv lo luôn việc tải Python 3.10 nếu máy bạn đang thấp hơn. Mở OrbStack (hoặc Docker Desktop) cho daemon chạy.

setup
$ brew install uv
$ open -a OrbStack          # hoặc mở Docker Desktop
$ docker info > /dev/null && echo "Docker OK"
  Docker OK
2

Khởi tạo config server (runtime Docker)

Sinh file ~/.sandbox.toml theo mẫu docker. Lần đầu, uv sẽ tải Python 3.10 + dependencies.

init-config
$ uvx opensandbox-server init-config ~/.sandbox.toml --example docker
  Downloaded cpython-3.10 · Installed 48 packages
  Wrote example config (docker) to ~/.sandbox.toml
3

Chạy server (chế độ local không key)

Config mẫu để trống api_key, nên cần xác nhận chạy chế độ insecure. Chỉ dùng ở local — đừng expose ra ngoài. Chạy ở một terminal riêng (nó chiếm tiền cảnh).

server · :8080
$ OPENSANDBOX_INSECURE_SERVER=YES uvx opensandbox-server
  Uvicorn running on http://127.0.0.1:8080
4

Kiểm tra server sống

Mở terminal khác. Swagger UI ở /docs; danh sách sandbox ban đầu rỗng.

healthcheck
$ curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8080/docs
  200
$ curl -s http://127.0.0.1:8080/v1/sandboxes
  {"items":[],"pagination":{"totalItems":0, ...}}
5

Cài & cấu hình CLI osb

Trỏ CLI về server local qua HTTP. Sau bước này lệnh osb có sẵn trên PATH.

cli config
$ uv tool install opensandbox-cli     # hoặc: uvx --from opensandbox-cli osb <cmd>
$ osb config init
$ osb config set connection.domain localhost:8080
$ osb config set connection.protocol http
  ✅ config saved
6

Pull sẵn image (khuyến nghị)

Tránh lỗi 401 tạm thời khi server pull execd đồng thời với image của bạn. Pull tay một lần là xong.

pre-pull
$ docker pull opensandbox/execd:v1.0.19
$ docker pull python:3.12
7

Tạo một sandbox

Tạo từ image python:3.12, TTL 30 phút. Để ý: id nằm ở field id trong JSON.

create
$ osb sandbox create --image python:3.12 --timeout 30m -o json
  {
    "id": "c8ffefe3-0ea0-4111-963c-b2369a100965",
    "image": "python:3.12",
    "status": "running"
  }
$ docker ps     # container thật + execd map cổng riêng
  sandbox-c8ffefe3...  python:3.12  Up  0.0.0.0:47713->8080 ...
8

Chạy lệnh / code bên trong

Thực thi trong sandbox và xác nhận nó cô lập khỏi host (không thấy /Users của macOS).

run
$ osb command run <id> -o raw -- python -c "import sys; print(sys.version)"
  3.12.13 ...
$ osb command run <id> -o raw -- ls /Users
  ls: cannot access '/Users': No such file or directory  → cô lập ✓
9

Dọn dẹp

Huỷ sandbox bằng kill (không có lệnh delete). Server thì dừng bằng Ctrl+C ở terminal đang chạy nó.

cleanup
$ osb sandbox kill <id>
  ○ terminated
$ # dừng server: Ctrl+C ở terminal chạy opensandbox-server
05 — Ổ gà thực tế

Những điểm dễ vấp (đã gặp)

Đây là các trục trặc gặp thật khi chạy trên máy macOS + OrbStack, và cách xử lý.

PY

Python 3.9 không đủ. macOS mặc định 3.9.6 < mức yêu cầu 3.10+. Không cần cài Python tay — uv tự tải bản 3.10 ngay ở bước init-config.

401

Pull execd báo 401 lúc tạo sandbox lần đầu. Đây là lỗi tạm thời của Docker Hub khi pull đồng thời, không phải image private. Chạy docker pull opensandbox/execd:v1.0.19 một lần trước (bước 6) là hết.

KEY

Server không chịu chạy nếu thiếu key. Config mẫu để trống api_key, nên phải đặt OPENSANDBOX_INSECURE_SERVER=YES. Chỉ dùng ở local; production thì đặt api_key thật.

JSON

Tên field là id, không phải sandboxId. Parse output -o json nhớ lấy .id — nếu không sẽ tưởng tạo thất bại trong khi sandbox đã chạy.

CLI

Huỷ sandbox là osb sandbox kill <id>. Không có lệnh delete. Các lệnh khác: create · list · get · kill · pause · resume · renew · metrics · endpoint · health.
